# Launch Playbook

Use this when you want the repo to land as a strong public project instead of just quietly existing on GitHub.

## What You Are Trying To Optimize For

There are two different outcomes:

- portfolio strength: strong README, proof, architecture clarity, release discipline, and clean engineering signals
- reach: a shareable demo, clear positioning, and a release people can understand in seconds

This repo already has a stronger engineering base than most early public projects. The highest-leverage next steps are distribution and proof, not more undirected feature work.

## Highest-Leverage Assets

### 1. Short demo video

Target:

- 30 to 60 seconds
- vertical or square
- no intro fluff

Show, in order:

1. connect screen
2. full pairing-style link paste
3. browserless in-app T3 session
4. photo upload flow
5. invalid certificate blocked cleanly

### 2. Strong release page

Every public release should make evaluation easy:

- signed APK
- checksum
- 3 screenshots
- `Tested on` block
- link to runtime evidence

### 3. One sharp public post

Do not lead with "I made an app." Lead with the user problem and the differentiator.

Use this shape:

- problem: using T3 from a phone through a browser is clunky
- solution: browserless Android companion app
- differentiators: pairing-link support, upload support, diagnostics, cert blocking, evidence-backed release
- CTA: APK + repo + screenshots

## Suggested Launch Sequence

1. Publish or refresh the release with evidence links.
2. Open 3 to 5 good starter issues.
3. Post the demo plus repo link.
4. Follow up with one engineering thread or post about what was technically hard.

## Good Starter Public Angles

- "I built a browserless Android companion for T3 Code"
- "This is what a serious WebView wrapper has to solve beyond just opening a URL"
- "Pairing links, upload support, diagnostics, and cert blocking for self-hosted mobile T3"

## Things That Increase Trust Fast

- real device screenshots
- a short demo clip
- visible release evidence
- a small architecture diagram
- a few focused public issues

## Things That Dilute The Launch

- shipping without a demo
- long vague posts with no visuals
- overclaiming beyond the evidence
- adding broad features that blur the repo identity

## Near-Term Execution List

- capture a real-device proof set
- capture a short release demo
- publish a clean release with `Tested on`
- open the starter issues from [STARTER-ISSUES.md](STARTER-ISSUES.md)
